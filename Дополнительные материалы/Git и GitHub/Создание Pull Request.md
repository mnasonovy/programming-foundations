# Создание Pull Request

**Pull Request** — это запрос на добавление ваших изменений из вашего fork репозитория в основной репозиторий курса.

В рамках курса Pull Request используется для сдачи домашних и практических заданий.

---

## Когда создаётся Pull Request

Pull Request создаётся после того, как вы:

1. Сделали **fork** репозитория.
2. Склонировали свой fork на компьютер.
3. Выполнили задание.
4. Создали commit.
5. Отправили изменения в свой fork через `git push`.

---

## Шаг 1. Открыть свой fork на GitHub

Перейдите на страницу своего fork репозитория.

Важно убедиться, что вы находитесь именно в своей копии проекта, а не в оригинальном репозитории.

В верхней части страницы должна быть пометка:

```
forked from <оригинальный репозиторий>
```

Если изменений пока нет, GitHub может показывать сообщение:

```
This branch is up to date with ...
```

Это означает, что ваш fork сейчас полностью совпадает с основным репозиторием.

![Fork репозитория без новых изменений](../Скриншоты/git%20&%20github/Создание%20Pull%20Request/img.png)



## Шаг 2. Проверить, что изменения отправлены в fork

После выполнения задания, создания commit и команды `git push` на странице fork появится сообщение примерно такого вида:

```
This branch is 1 commit ahead of ...
```

Это означает, что в вашем fork есть изменения, которых ещё нет в основном репозитории.

GitHub также может показать кнопку:

```
Open pull request
```

Нажмите её.

Если кнопка не появилась, можно нажать:

```
Contribute → Open pull request
```
![Fork на один commit впереди основного репозитория](../Скриншоты/git%20&%20github/Создание%20Pull%20Request/img_1.png)

---

## Шаг 3. Проверить направление Pull Request

Откроется страница **Comparing changes**.

На этой странице нужно внимательно проверить направление Pull Request.

Должно быть так:

```
base repository: основной репозиторий курса
base: main

head repository: ваш fork
compare: ветка с вашим заданием
```

Иными словами, изменения должны идти:

```
из вашего fork → в основной репозиторий
```

Если вы выполняли задание прямо в ветке `main` своего fork, то в поле `compare` будет выбрана ветка `main`.

Для учебных заданий лучше создавать отдельную ветку под каждое задание, но на простом примере допустимо использовать `main`.

Также проверьте сообщение:

```
Able to merge.
```

Оно означает, что GitHub не видит конфликтов и Pull Request можно создать.

---

## Шаг 4. Заполнить название Pull Request

В поле **Add a title** укажите короткое и понятное название.

Примеры:

```
Добавлено первое домашнее задание
```

```
Практическое задание по модулю 1
```

```
Задание 1.2 — первая программа на C
```

Название должно сразу показывать, что именно отправляется на проверку.

---

## Шаг 5. Заполнить описание Pull Request

В поле **Add a description** кратко опишите, что было сделано.

Пример описания:

```
Выполнено первое домашнее задание по уроку 1.2 — Первая программа на C.

Что сделано:
- создан проект в Visual Studio 2022;
- написана первая программа на C;
- добавлен скриншот результата запуска.

Проверка:
- программа запускается без ошибок;
- в консоли отображается сообщение.
```

После заполнения названия и описания нажмите:

```
Create pull request
```

![Заполнение названия и описания Pull Request](../Скриншоты/git%20&%20github/Создание%20Pull%20Request/img_2.png)

## Шаг 6. Проверить созданный Pull Request

После создания Pull Request откроется его страница.

На ней можно увидеть:

- название Pull Request;
- статус `Open`;
- из какого репозитория и ветки идут изменения;
- в какой репозиторий и ветку они предлагаются;
- список commits;
- вкладку **Files changed**;
- блок проверки конфликтов.

Если отображается сообщение:

```
No conflicts with base branch
```

значит конфликтов нет.


![Созданный Pull Request](../Скриншоты/git%20&%20github/Создание%20Pull%20Request/img_3.png)

> Важно: студенту не нужно нажимать **Merge pull request**.  
> Pull Request создаётся для проверки. Решение о принятии изменений принимает преподаватель или владелец основного репозитория.

---

## Шаг 7. Посмотреть список Pull Request

Все Pull Request можно найти во вкладке:

```
Pull requests
```

Там отображаются открытые и закрытые Pull Request.

Если Pull Request уже проверен и закрыт, он будет находиться в разделе:

```
Closed
```

![Список закрытых Pull Request](../Скриншоты/git%20&%20github/Создание%20Pull%20Request/img_4.png)

---

## Шаг 8. Исправить замечания, если они есть

Если проверяющий оставил замечания, не нужно создавать новый Pull Request.

Исправления добавляются в ту же ветку, из которой был создан Pull Request.

Внесите исправления, затем выполните:

```
git add .
git commit -m "Исправил замечания"
git push
```

После команды `git push` Pull Request обновится автоматически.

---

## Частые ошибки

| Ошибка | Что не так | Как исправить |
|---|---|---|
| Pull Request создан не из fork | Изменения должны идти из вашей копии репозитория | Проверьте поле `head repository` |
| Pull Request направлен не в основной репозиторий | Изменения могут уйти не туда | Проверьте поле `base repository` |
| В Pull Request попали лишние файлы | Перед commit не проверили изменения | Используйте `git status` перед `git add .` |
| После замечаний создан новый Pull Request | Проверяющему сложнее отслеживать исправления | Исправляйте ту же ветку и делайте `git push` |
| Название Pull Request слишком общее | Непонятно, какое задание отправлено | Указывайте номер модуля или урока |
